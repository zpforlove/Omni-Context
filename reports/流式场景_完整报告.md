# 流式场景完整报告：增量 Context 增益曲线 + 实时门控注入

> **命题**：①把音频换成流式输入（模型在 t 时刻只能听到前 t 秒、线索也只能由前 t 秒算出），"只给线索不给答案"的 Context 在 1–2s 短音频时是否已能显著提升 Omni 模型？还是必须 offline 拿到完整长音频才有用？②能否用一个实时注入检测器做门控——干净对话段不打扰模型、只在"多人重叠+强噪声"段启动线索注入？
> 场景：S2 重叠+噪声（最难）。模型：Qwen3-Omni-30B / MiniCPM-o-8B / Ming-flash-104B-MoE（基座，未加训练 LoRA——考察推理时增益）。
> 规模：曲线 55 样本×7 前缀×3 条件×3 模型（5,235 次推理）+ 门控 30 流×5 策略×3 模型 + 探针 360 窗。
> 代码 `code/stream_*.py`，原始结果 `results/stream_*`，本报告所有图表可由 `stream_plot.py` / `stream_aggregate2.py` 一键再生。

---

## 一、结论先行（大白话）

1. **1–2 秒的音频太短，线索"还没长出来"**：线索是自动工具（SepFormer 分离+ASR）从音频里算的，1s 音频只能算出 **0 个关键词**、2s 约 2 个、3s 约 5 个、8s 约 15 个、全量约 17 个。所以"超短音频就显著提升"不成立——**不是方法失效，是信息量物理上不够**。弱模型（Ming）在 1–2s 时甚至被空线索框架拖累（−11.2/−5.2）。
2. **线索从 3–4 秒开始"成熟"，难样本增益跟着长出来**：真流式下 Qwen3 在 2–4s 区间整体增益转正（3s 处全部 +2.9、难1/3 **+8.2**）；线索成熟度曲线上，**6 秒音频算出的线索已拿到全量线索 70–80% 的增益**（Ming 难1/3：6s +16.7 vs 全量 +21.2）。
3. **所以答案是"中间态"**：Context 增益**不必等 offline 全量音频，但也不是 1–2s 就能起效**——线索需要 **3–6 秒**的积累；一旦成熟，流式增益与 offline 同向、同规律（集中在弱模型+难样本）。
4. **门控可行且省 65% 注入量，但"线索怎么写"决定成败**：发现并修复**注意力虹吸**——带时间窗的门控线索使干净段转写召回暴跌（Qwen3 97.6→73.5）；线索里**加一句"仍需完整转写全程"**（gated_v2）后干净段满血恢复、Qwen3 整体 cpWER 还做到五策略最优（36.2 vs 基线 38.1）。
5. **"模型自己当门控开关"零样本还不行，但地基已在**：CLEAN 识别 90–100%，COMPLEX 仅 0–57%（边界保守）——GRPO 强化学习正好补这个空白（方案见第七节）；MiniCPM 单窗判定 0.30s，实时性达标。

## 二、实验设计

### 2.1 实验一：增量 Context 增益曲线
- **数据**：SparseLibriMix2_noisy（双人重叠+WHAM 噪声，SNR 0/5/10）中时长≥8s 的全部 **55 条**；前缀 t∈{1,2,3,4,6,8,full}s → 385 个截断音频 + 385 份前缀线索。
- **流式模拟（关键诚实点）**：t 时刻的线索**只能由 audio[0:t] 算出**——SepFormer 分离→双路 Mega-ASR→部分打乱关键词（与 offline 管线相同、零泄漏门禁不变）。实测线索信息量随音频积累增长：

| 前缀 | 1s | 2s | 3s | 4s | 6s | 8s | full |
|---|---|---|---|---|---|---|---|
| 平均关键词数 | **0.0** | 2.0 | 5.0 | 7.2 | 11.8 | 15.1 | 16.9 |

- **两条曲线分离两个混杂因素**：
  - **曲线A（真流式）**：模型听截断音频，比三条件：baseline（无线索）/ agsc-stream（前缀线索）/ agsc-offline（全量线索=oracle 上界）；
  - **曲线B（线索成熟度）**：模型听**完整音频**，线索由前缀算——单独回答"线索要积累几秒才有用"，排除"截断音频听不到尾部"对分数的稀释。
- **指标**：cpWER（对完整 label 评分，截断导致的删除两条件同等承担，对比公平）；GCG=baseline−agsc；难 1/3 切片（按各样本 baseline 难度排序）。

### 2.2 实验二：流式实时门控
- **合成流**：[干净单人 4s][双人重叠+噪声 4s][干净单人 3s] × **30 条**（拼接构造→复杂区真值位置免标注）。干净段转写参考用 Mega-ASR 伪标注（干净音频上近金标，如实标注）。
- **信号检测器**：逐 1s 窗 pyannote 重叠检测（活跃说话人≥2）→ 滞回状态机平滑成连续 `<ctx_on>/<ctx_off>` 区间。
- **端到端五策略**：baseline 无线索 / always 整流盲算盲注入 / gated 检测器区间线索+时间窗说明 / gated_gt 真值区间（oracle 检测器上界）/ **gated_v2** 真值区间+"仍需完整转写全程"提醒。
- **评分**：4 参考（干净A+说话人1+说话人2+干净B）置换不变 cpWER + **干净段/复杂段词召回分离**（直接验证"门控保干净段、增复杂段"）。
- **一 token 探针**：从流中切 1s/2s 窗（干净/复杂各半，每模型 120 窗），模型输出单词 COMPLEX/CLEAN，测准确率+单窗延迟。

### 3.1 曲线A：真流式（截断音频 + 前缀线索）

**主图（难样本切片）怎么读**：横轴=模型听到的音频长度（10=全量），纵轴 cpWER 越低越好；只看每个模型最难的 1/3 样本（n=18，模型真正失败的地方）。**Ming 面板从 3s 起出现整片绿色增益区**（−3.4 → −8.3 → 全量 −12.3），Qwen3 在 3s 处有 −8.2 的增益口袋（它只听到 3 秒、自身 cpWER 88 时，5 个关键词就能拉低 8 个点）。与 offline 规律完全一致：**推理时线索的增益在失败点真实存在且不小**。

## 四、实验二结果：流式实时门控

### 4.1 信号检测器（30 条流，逐秒）
| 指标 | 数值 |
|---|---|
| 逐秒 Precision / Recall / F1 | 0.549 / 0.517 / **0.532** |
| 触发延迟（复杂区开始→首次注入） | **1.72s** |
| 干净段豁免率（干净秒未被打扰比例） | **75%** |
| 注入量节省（vs 全程注入） | **65.1%** |

检测器是廉价信号链路（pyannote 逐窗重叠+滞回平滑），F1 0.53 够用不优秀——下图显示它正是 gated 与 gated_gt（oracle）差距的主因。

### 4.2 端到端五策略

![门控五策略：上排 cpWER（黑框=该模型最优），下排干净段召回（虚线=基线水平）](results/fig_gate_strategies.png)

**怎么读这张图**：上排看整体准确（越低越好，黑框=最优），下排看"干净段是否被打扰"（虚线=不注入时的基线水平，低于虚线=被打扰）。三个故事一眼可见：
1. **下排中间三根矮柱（gated/gated_gt 的橙、紫柱）就是"注意力虹吸"**——朴素的时间窗线索让模型只顾复杂段、干净段被静默丢弃（Qwen3 召回 97.6→73.5，MiniCPM 最惨 →57.3）；

## 五、与 offline 结论的对照（四条规律全部复现 + 新增一条）
| offline 规律 | 流式验证结果 |
|---|---|
| 增益集中在模型失败点 | ✅ 曲线B 虚线（难1/3）全面正、实线（全部）贴零 |
| 前置能力+Context 设计决定成败 | ✅ 短前缀线索空→无增益；注意力虹吸→一句措辞翻盘 |
| 越弱的模型收益越大 | ✅ 成熟度难1/3：Ming +21.2 > MiniCPM +15.7 > Qwen3 |
| 干净场景注入有害→需门控 | ✅ 端到端实测盲注入/坏门控伤干净段，门控+精准措辞修复 |
| **（新）线索措辞一句话改变门控成败** | gated→gated_v2：Qwen3 cpWER 48.2→36.2、召回 73.5→97.6 |

## 六、诚实记录的负结果与边界
1. **1–2s 超短前缀无增益是物理边界**（线索 0–2 词），突破需"前置组件低延迟化"（流式分离/流式 ASR），而非 prompt 工程。
2. **检测器 F1 仅 0.53**：pyannote 在 1s 窗+拼接流上重叠检测偏弱（干净区偶发误报）；触发延迟 1.72s 意味着复杂段头 1–2s 必然漏注入。
3. **Ming 在混拼流上全策略负增益**：对 prompt 注入敏感（同 offline），其流式价值在难样本切片与训练内化，不在推理时注入。
4. **gated_v2 对 MiniCPM 的副作用**（啰嗦→插入错误）说明提醒语也需按模型调优——没有万能 prompt。
5. 干净段参考是 Mega-ASR 伪标注（干净音频上近金标）；截断前缀的绝对 cpWER 含不可避免的删除项，但所有条件同等承担，**对比（GCG）有效**。

## 七、GRPO 内化门控：设计方案（下一阶段）
**目标**：把"何时要脚手架"从外置检测器变成模型自身能力——流式解码中自己发射门控 token。
1. **动作空间**：词表加 `<ctx_on>` / `<ctx_off>`；每消费 1–2s 音频 chunk 可发射，`<ctx_on>` 触发线索链路并把关键词追加进上下文。
2. **奖励**（GRPO 组内相对优势，G=8 rollouts）：r = r_下游 + α·r_门控 − λ·r_成本。r_下游=最终转写 cpWER 相对组内均值的改善（主奖励，保证门控为听准服务）；r_门控=逐秒决策与真值复杂区的 F1（合成流真值免标注）；r_成本=注入秒占比（已证盲注入有害且费时）。
3. **冷启动**：合成流真值轻量 SFT 教发射格式 → GRPO 调策略。一 token 探针给出零样本下界（COMPLEX 33–57%）：表征里有信号、边界保守——正是 RL 能调的对象。
4. **数据免标注**：拼接流复杂区位置天然已知，奖励全程序化——延续"管线自动合成"路线。
5. **工程（driver 470）**：避开 trl 新版依赖，LoRA 策略层+手写组内优势（复用本项目手动 loss 训练器模式），MiniCPM 单卡起步（0.3s/窗也最适合当哨兵）。规模 ~3500 流×8 rollouts ≈ 1–2 天单卡。
